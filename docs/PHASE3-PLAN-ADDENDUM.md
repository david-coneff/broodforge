## Phase 3.L — OpenBao Secrets Broker

**Goal:** Replace `keepassxc-cli` as the machine-facing secrets broker with OpenBao
(open-source Vault fork), while keeping KeePass as the human-facing encrypted
credential store. Adds policy enforcement, TTL, audit logging, and RBAC that KeePass
cannot provide.

**Design decisions:**

- **Topology:** OpenBao runs as a systemd service on the governance VM (same host as
  `broodforge_dashboard.py`). It is not exposed outside the Headscale overlay network.
- **Auto-unseal:** Use OpenBao Transit auto-unseal backed by a second OpenBao instance,
  OR Shamir 3-of-5 unseal keys split across KeePass master DB + two offline USB keys.
  Decision required before implementation.
- **Bootstrap ceremony:** `scripts/forge-init-openbao.sh` seeds OpenBao from KeePass
  on first install. Operator unlocks KeePass master, script reads each child-DB entry
  via `keepassxc-cli`, writes to OpenBao path. Ceremony is one-time and operator-gated.
- **Policy mapping:**
  - `forge-autonomous` child DB → OpenBao policy `forge/autonomous/*` (no human TTY)
  - `forge-spawn` child DB → OpenBao policy `forge/spawn/*` (spawn-scoped secrets)
  - `forge-migrate` child DB → OpenBao policy `forge/migrate/*` (migration-scoped)
  - Each policy limits GET to its own path prefix; no cross-policy reads.
- **`child://` reference scheme** preserved. `kdbx_get_child()` in `lib/forge-lib.sh`
  is replaced by `_bao_get_secret()` which calls `bao read -format=json <path>`.
  The `child://db/group/entry` path maps to `forge/<db>/<group>/<entry>`.
- **TOTP:** OpenBao has a TOTP secrets engine. `kdbx_totp()` in `forge-lib.sh` is
  replaced by `_bao_totp()` which calls `bao read forge/totp/code/<account>`.
  Existing TOTP secrets are migrated to OpenBao TOTP engine during bootstrap.
- **Key rotation:** Same 7-step ceremony; after operator confirms service accepts new
  secret, `bao kv put` updates the OpenBao path, then `forge-sync-credentials.sh`
  updates KeePass as the audit record (not the live broker).

**Files to create/modify:**
- `proxmox-bootstrap/openbao_manager.py` — Python API client: init, unseal, policy
  apply, secret read/write, TOTP engine config, audit log query
- `scripts/forge-init-openbao.sh` — Bootstrap ceremony: seed from KeePass → OpenBao
- `scripts/forge-openbao-unseal.sh` — Manual unseal helper (emergency use)
- `config/openbao-policies/` — HCL policy files for each forge role
- `lib/forge-lib.sh` — Replace `kdbx_get_child()` and `kdbx_totp()` with OpenBao calls;
  keep KeePass helpers for operator-facing credential lookups
- `docs/ARCHITECTURE.md` — Add OpenBao to secrets architecture section
- `docs/ROADMAP.md` — Mark 3.L complete when done

**Dependencies:** Phase 1.P (credential hierarchy) must be complete. OpenBao binary
must be available on governance VM.

**Operator decisions needed before implementation:**
1. Auto-unseal strategy (Transit vs. Shamir split)
2. Whether KeePass remains writable after bootstrap or becomes read-only audit log

## Phase 3.M — In-Page Markdown Source Editor

**Goal:** Each HTML page generated by `proxmox-bootstrap/md_to_html.py` should embed
its own markdown source and expose an in-page editor so operators can fix AI-generated
errors directly without a full re-prompt cycle.

**This is distinct from the existing inline block editor** (✎ per paragraph, edits
rendered HTML). This editor exposes and saves the raw markdown source.

**Design:**

**Tier A — Embed + textarea (no server dependency):**
- `md_to_html.py` embeds raw markdown as `const _MD_SOURCE = \`...\`` JS variable in
  every generated page.
- Toolbar gets `✎ Edit Source` toggle button (right of theme toggle).
- Editor pane: full-width `<textarea id="bf-source-editor">` replaces `#bf-doc-body`
  when active. Styled with `--code-bg` / `--code-text`, monospace font, `white-space:pre`.
- Lint pass (JS, inline): on textarea `input`, highlight lines with unknown `@directive`
  syntax or malformed `child://` refs using `::before` markers in a parallel gutter div.
- `⬇ Download .md` button: `Blob`/`URL.createObjectURL` download of textarea content.
  Available even without dashboard.

**Tier B — Save & Regenerate (requires dashboard):**
- `broodforge_dashboard.py` gains endpoint `POST /api/docs/edit` accepting
  `{id: string, source: string}`. Handler writes the `.md` file atomically
  (write to `.tmp` then `os.replace()`), then calls `regenerate_docs.py --id=<id>`.
- Toolbar `💾 Save & Regenerate` button: POSTs to `http://localhost:PORT/api/docs/edit`.
  On 200: reloads the page. On network error: falls back to download-only mode with a
  toast: "Dashboard not running — download .md to save manually".
- Dashboard port is embedded at page-generation time as `const _DASHBOARD_PORT = NNNN`.

**Files to create/modify:**
- `proxmox-bootstrap/md_to_html.py` — Embed `_MD_SOURCE`, add source editor toggle,
  lint gutter, download button, save-and-regenerate button
- `proxmox-bootstrap/broodforge_dashboard.py` — Add `POST /api/docs/edit` endpoint
- `proxmox-bootstrap/regenerate_docs.py` — Already supports `--id`; verify atomic write
- All HTML docs — regenerate after md_to_html.py update

**Dependencies:** Tier A has none. Tier B requires dashboard to be running.

## Phase 3.N — TOTP QR Code in HTML Pages

**Goal:** Walkthrough docs that configure TOTP-protected services should show a
scannable QR code so operators can add to their authenticator app without manually
typing a base32 secret.

**Design constraints:**
- TOTP secret must NOT be stored in HTML source (HTML files are in git).
- No external CDN dependencies (anti-pattern per DESIGN-SCHEMA.md).
- QR codes must be printable: high contrast, minimum 200px, white background.

**Approach:**

New directive `@totp-qr[Label|account|issuer]` in `md_to_html.py`. Paired with an
`@credential[TOTP Secret|TOTP_SECRET]` field on the same page.

- When the user fills the credential field with the base32 TOTP secret, JS builds the
  `otpauth://totp/<issuer>:<account>?secret=<VALUE>&issuer=<issuer>` URI and renders
  a QR code into a `<canvas>` element.
- QR library: inline a compact pure-JS QR encoder (~6KB minified) as a JS constant
  in `_JS` section of `md_to_html.py`. No CDN. Use a permissively-licensed minimal
  implementation (e.g. port of qrcode-generator).
- QR canvas: 250×250px, `background:#ffffff` forced (print-safe), `color:#000000`.
  A `☀ Print QR` button opens a minimal print-only window with just the canvas.
- `👁 Peek` toggle: shows the raw base32 secret as copyable text below the QR.
  Secret lives only in sessionStorage (same as other `@credential` fields).
- QR regenerates on every `input` event of the linked credential field.
- If credential field is empty: shows a placeholder with dashed border and
  label "Fill TOTP Secret field above to generate QR".

**Files to create/modify:**
- `proxmox-bootstrap/md_to_html.py` — Add `@totp-qr` directive, inline QR library,
  JS wiring for credential field → canvas render, print helper
- `docs/DESIGN-SCHEMA.md` — Document `@totp-qr` directive
- Any walkthrough docs that set up TOTP-protected services — add `@totp-qr` where
  appropriate (e.g. Authentik, KeePass MFA setup)

**Dependencies:** Phase 3.M (editor) is independent. No dependency on OpenBao.

## Phase 3.O — TOC Tiered Numbering

**Goal:** The auto-generated Table of Contents should mirror the full collapsible
section hierarchy with numeric-only tiered numbering (1, 1.1, 1.1.1). No roman
numerals, no letters.

**Current state:** TOC is h2-only, flat, one level deep.

**Design:**

- Walk the full heading tree (h2, h3, h4) collected during md_to_html.py rendering.
- Assign counters hierarchically: h2 increments the top-level counter (resets h3/h4);
  h3 increments second-level; h4 increments third-level.
- TOC renders as nested `<ul>` with counter-prefix `<span class="bf-toc-num">`:
  ```
  1 · Introduction
    1.1 · Background
    1.2 · Scope
  2 · Installation
    2.1 · Prerequisites
      2.1.1 · Hardware
  ```
- The `<details id="bf-toc">` block is itself collapsible at each tier: each h2 entry
  that has h3 children is wrapped in a nested `<details class="bf-toc-section">` with
  the h2 text as `<summary>`.
- Number prefix also appears as a small `<span class="bf-section-num">` badge in the
  `<summary>` of each `.section` and `.subsection` element (right-aligned, `--muted` color).
- **Active section highlighting:** `IntersectionObserver` watches each
  `details.section > summary` and `details.subsection > summary`. When a section
  scrolls into the viewport, its TOC entry receives class `bf-toc-active`
  (`color: var(--accent); font-weight:600`). Smooth, debounced (16ms).
- TOC is generated only when ≥ 3 h2 sections exist (same threshold as now).

**Files to create/modify:**
- `proxmox-bootstrap/md_to_html.py` — Rewrite TOC generation: hierarchical walker,
  counter assignment, nested `<ul>/<details>` render, section number badges,
  IntersectionObserver JS, `bf-toc-active` CSS rule
- `docs/DESIGN-SCHEMA.md` — Update TOC section
- All HTML docs — regenerate after md_to_html.py update

**Dependencies:** Can be implemented independently of 3.M and 3.N.

## Phase 3.P — Setup Guide Markdown Migration

**Goal:** `docs/SETUP-GUIDE.html` is hand-authored (~150KB) with no `.md` companion.
Migrate it to be authored in markdown and generated via `md_to_html.py`, matching the
look and feel of all other doc pages.

**Steps:**

1. **Audit** `docs/SETUP-GUIDE.html` for schema element opportunities:
   - Step-by-step commands → `@field` / `@dir` for paths, IPs, hostnames
   - Passwords and keys → `@credential`
   - Yes/no configuration choices → `@check` or `@radio`
   - Multi-step verification commands → `@parse` (console output extractor)
   - TOTP setup steps → `@totp-qr` (after Phase 3.N)

2. **Author** `docs/SETUP-GUIDE.md` using md_to_html.py schema. Use collapsible
   sections (`##`) for major setup phases and subsections (`###`) for sub-steps.
   Preserve all existing content; do not silently drop steps.

3. **Update** `proxmox-bootstrap/doc-manifest.json`:
   - Remove `"handAuthored": true` from the setup-guide entry
   - Add `"source": "docs/SETUP-GUIDE.md"`

4. **Regenerate** `docs/SETUP-GUIDE.html` from the new `.md` source.

5. **Verify** visual parity: open both old and new HTML side by side and confirm
   all sections, commands, and tables are present.

6. **Commit** `.md` source + regenerated `.html` together. The `.html` is now
   generated, not hand-authored.

**Note:** SETUP-GUIDE is the largest doc in the repo. The migration will likely
reveal missing schema affordances (e.g. fields that would benefit from `@parse`
but weren't possible in hand-authored HTML). Log these as follow-up items.

**Files to create/modify:**
- `docs/SETUP-GUIDE.md` — New canonical source (author from scratch by extracting
  content from existing HTML)
- `docs/SETUP-GUIDE.html` — Regenerated from .md
- `proxmox-bootstrap/doc-manifest.json` — Remove handAuthored, add source path

**Dependencies:** Phases 3.N (TOTP QR) should land first if setup guide includes
TOTP setup steps. Phase 3.M (source editor) is useful but not required.
