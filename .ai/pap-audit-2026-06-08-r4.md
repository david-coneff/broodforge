## Part B — New Findings

---

### R4-001 · MEDIUM · Session file does not persist FORGE_KDBX_PATH; kdbx_get still fails silently in phase-05/06

**File:** `proxmox-bootstrap/forge_scripts.py` — `FORGE_KEEPASS_GATE_SH` definition, session resume branch (`forge_scripts.py:113–118`)

**Root cause analysis:**

Each forge phase runs as a separate bash subprocess of forge.sh:
```bash
bash "$SCRIPT_DIR/$script"  # forge.sh:252
```

Phase-03's subprocess exports `FORGE_KDBX_PATH` after unlock:
```bash
export FORGE_KDBX_PATH  # inside forge_keepass_gate, phase-03's process
```

But `FORGE_KDBX_PATH` is only exported within phase-03's process tree. Phases 05 and 06 are children of forge.sh (not phase-03), so they do not inherit phase-03's exported variables. When phases 05/06 source `forge-keepass-gate.sh`, it initializes `FORGE_KDBX_PATH=""`.

The session resume branch attempts to restore it:
```bash
KEEPASS_MASTER_PASSWORD="$(cat "$_FORGE_SESSION_FILE")"
# FORGE_KDBX_PATH is exported; restore if empty (e.g. first source in this shell)
[ -z "$FORGE_KDBX_PATH" ] && FORGE_KDBX_PATH="${FORGE_KDBX_PATH:-}"
```

The comment is factually incorrect — `FORGE_KDBX_PATH` is **not** available in the phase-05/06 environment. The assignment `FORGE_KDBX_PATH="${FORGE_KDBX_PATH:-}"` is a no-op: it assigns an empty variable to itself.

**Consequence:** After resuming from the session file, `FORGE_KDBX_PATH` is empty. `kdbx_get` calls:
```bash
printf '%s\n' "$KEEPASS_MASTER_PASSWORD" | \
  keepassxc-cli show -q -a Password "" "<key-path>" 2>/dev/null
```
`keepassxc-cli` fails (empty database path), and `2>/dev/null` suppresses the error. The command substitution `$(kdbx_get ...)` returns an empty string. The fix that correctly propagates the password does not deliver working credentials to phases 05/06.

**Current impact (still latent):** Same as described in R3-004 — phase-05's kdbx_get is inside the never-reached ansible branch; phase-06's empty kdbx_get exits 2 via the credentials-missing guard. Observable behavior is unchanged. However, the intended fix for the "operator added credentials but forge still loops" scenario is not yet complete.

**Correct fix:** Store `FORGE_KDBX_PATH` in the session file alongside the password (e.g., line 1 = path, line 2 = password), and restore it in the resume branch. Alternatively, always call `forge_keepass_find_db` when resuming (the embedded kdbx lookup is idempotent and does not prompt if a kdbx file is present in `$SCRIPT_DIR/kdbx/`).
