## Part B — New Findings

### N-001 · HIGH · Phase-05 k3s token persistence crashes with `KeyError` at runtime

**File:** `proxmox-bootstrap/forge_scripts.py:601–620`

The Python heredoc added to phase-05 to persist k3s join tokens (F-011/F-020 fix) reads tokens from `os.environ`:

```python
state["k3s"]["worker_join_token"] = os.environ["_worker_token"]
state["k3s"]["server_join_token"] = os.environ["_server_token"]
```

`_worker_token` and `_server_token` are bash shell variables set without `export`:

```bash
_worker_token="$(kdbx_get 'Infrastructure/k3s/worker-join-token')"
_server_token="$(kdbx_get 'Infrastructure/k3s/server-join-token')"
```

In bash, unexported shell variables are not inherited by child processes. `python3 - <<PYEOF` spawns Python as a child process; `os.environ["_worker_token"]` raises `KeyError`. Python exits 1. Since phase-05 has `set -euo pipefail`, the script aborts, causing forge.sh to see "FAIL phase-05-k3s.sh" after ansible has already successfully installed k3s.

**Compounding effect:** `checkpoint_done "$step"` is called after the heredoc, so phase-05 is never checkpointed. Every retry of `bash forge.sh --from 5` re-runs ansible (idempotent, harmless) then hits the same `KeyError` — an infinite failure loop.

**Result:** The F-011/F-020 fix is structurally inoperable. k3s join tokens are never written to bootstrap-state.json. Spawn plans continue to embed `None` for both tokens, and all broodling k3s joins will fail.

**Fix:** Either (a) export the variables (`export _worker_token="..."`) before the heredoc, or (b) pass the token values via a temp file and read them in Python with `open(path).read().strip()`, consistent with the pattern already used for the ansible k3s vars file.

### N-002 · HIGH · FORGE_INCOMPLETE banner unreachable in standard `bash forge.sh` run

**File:** `proxmox-bootstrap/forge_scripts.py:213–277`

When `bash forge.sh` runs with phases 04/05 being NOT_IMPLEMENTED, the intended flow is:
1. Phase-04 exits 2 → `_forge_incomplete=1`, loop continues
2. Phase-05 exits 2 → `_forge_incomplete=1`, loop continues
3. After all phases complete, the `if [ "$_forge_incomplete" -eq 1 ]` check shows the comprehensive FORGE_INCOMPLETE banner

**What actually happens:** Phase-06 runs next, checks for Forgejo credentials, finds none, and exits 1. The forge.sh loop sees exit code 1 (not 2):

```bash
else
  echo "[forge] FAIL $script (exit $_phase_exit) — aborting." >&2
  echo "[forge] Re-run: bash forge.sh --from ${{phase_num}}" >&2
  exit 1
fi
```

The FORGE_INCOMPLETE banner (line 244–265) is never reached. The operator sees `[forge] FAIL phase-06-gitops.sh (exit 1)` — not the 5-step instruction list that explains what manual work is needed.

Phase-06's own error message does provide partial guidance (Forgejo credential steps only), but omits the k3s token population steps that are also required before resuming from phase 6.

The FORGE_INCOMPLETE banner is only reachable if the operator runs `bash forge.sh --from 7` or later, skipping phase-06. It is never shown during a standard end-to-end forge run on a system where phases 04/05 have not been manually completed.

**Fix:** Phase-06 should exit 2 (NOT_IMPLEMENTED) instead of 1 when credentials are missing because upstream phases have not been completed — or the forge.sh loop should skip downstream phases when `_forge_incomplete=1`.

### N-004 · MEDIUM · Phase 1.J pre-generated spawn media: authorization records never populated by any CLI

**File:** `proxmox-bootstrap/_image_builder.py:455–502`; `proxmox-bootstrap/generate-bootstrap-image.py`

`_image_builder.py` implements:
- `build_pregenerated_spawn_media_record()` — generates a spawn-media credential + authorization record
- `record_pending_join_authorization()` — appends the record to bootstrap-state.json's `pending_join_authorizations` list

Both are well-tested and correctly implemented. However, neither is called by `generate-bootstrap-image.py` (the primary CLI for Phase 1.H/1.J) or any other CLI. The CLI uses `generate_install_passphrase()` directly — a regular install passphrase with no paired authorization record.

The `pending_join_authorizations` list in bootstrap-state.json will always be empty unless an operator writes custom code calling these library functions. `authorize-spawn-media-join.py` is the correct human-gate CLI, but it has nothing to authorize since no CLI creates records.

The ROADMAP.md Phase 1.J scope item "Pre-generated spawn-media credentials" is marked `[x]`. The library implementation is complete, but the end-to-end CLI workflow for the "build spawn media → create authorization record → human authorizes" sequence is unwired. An operator following standard tooling cannot exercise this feature.

1. **N-001** — Export `_worker_token`/`_server_token` (or use a temp file) in phase-05. One-line fix, unblocks the entire k3s token → spawn pipeline. The current code loops forever on a deployed system with a real ansible inventory.

This makes the FORGE_INCOMPLETE banner the outcome of a standard forge run on a system without phases 04/05 done, replacing the confusing "FAIL phase-06" error.

3. **N-003** — Add a "WAN mode prerequisites" section to NODE-SPAWNING.md covering: Headscale auth key generation (`headscale preauthkeys create --expiration 1h`), KeePass storage path, and `--wan-auth-key` CLI flag.

4. **N-004** — Either wire `build_pregenerated_spawn_media_record()` into `generate-bootstrap-image.py` (add a `--spawn-media` flag that changes the passphrase generation path and emits the authorization record), or add a standalone `generate-spawn-media.py` CLI that owns the full workflow. Until then, mark the Phase 1.J scope item for "pre-generated spawn-media credentials" as partially complete.
