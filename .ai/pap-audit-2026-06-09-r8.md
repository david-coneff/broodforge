All six `.sh` files open with `set -euo pipefail`. Variable expansions are double-quoted throughout. No obvious portability issues observed.

### ruff / mypy (manual structural review)
All Python files use `from __future__ import annotations`. Type annotations are present on all public functions. `Optional[...]` used consistently from `typing`. `# type: ignore[union-attr]` comments used narrowly on `importlib` dynamic dispatch only. No bare `except:` — all broad catches use `except Exception`. The `# noqa: BLE001` on the broad migration-failure catch in `run_migrations()` is correct (intentional).

### pytest --cov (manual structural review)
`test_migration_manager.py` provides extensive coverage of `migration_manager.py` and `bootstrap_state.py`. Coverage gaps: `package_verifier.py` has zero test coverage. See **R8-003**.

### R8-001 · OBSERVATION · `_load_schema_version()` silently swallows all exceptions

**Pattern:** #8 Silent Degradation
**Files:** `proxmox-bootstrap/migration_manager.py:79`, `proxmox-bootstrap/bootstrap_state.py:55`, `proxmox-bootstrap/package_verifier.py:136`

**Description:**
All three files contain an identical `_load_schema_version()` function with this exception handler:
```python
try:
    ...load version.py via importlib...
except Exception:
    pass
return "2026-06-09_00-00-00_0000000"
```

If `version.py` is missing, malformed, or raises an unexpected error, the exception is silently swallowed and the hardcoded baseline version is returned with no diagnostic output. On a healthy deployment this is harmless (the fallback IS the correct baseline). On a deployment with a corrupted `version.py`, the code continues silently with the wrong version assumption.

**Risk:** Low. The fallback value is the baseline version, not a dangerously wrong default. But the silent swallow makes `version.py` corruption invisible.

**Recommended fix:** Log at WARNING level before returning the fallback:
```python
except Exception as exc:
    import logging as _logging
    _logging.getLogger(__name__).warning(
        "Cannot load SCHEMA_VERSION from version.py (%s) — using baseline fallback", exc
    )
return "2026-06-09_00-00-00_0000000"
```

**Additional observation:** `_load_schema_version()` is identically defined in three files. A module-level shared utility (e.g., `_version_loader.py`) would eliminate the triplication and ensure any fix propagates to all three callers.

### R8-002 · DEFECT · `_parse_args()` raises unhandled `IndexError` on malformed CLI invocation

**Pattern:** #10 Happy Path Only
**File:** `proxmox-bootstrap/migration_manager.py:441-455`

**Description:**
```python
if arg == "--state-dir":
    args["state_dir"] = argv[i + 1]   # IndexError if --state-dir is the final token
    i += 2
elif arg == "--migrations-dir":
    args["migrations_dir"] = argv[i + 1]  # same
    i += 2
```

If an operator runs `python3 migration_manager.py --state-dir` (omitting the path), `argv[i + 1]` raises `IndexError: list index out of range`. This produces an unhandled traceback instead of a usage message.

The current primary caller (`forge-migrate.sh`) always provides the value, so this doesn't affect the scripted happy path. But direct invocation with a typo gives a confusing error.

**Recommended fix:**
```python
if arg in ("--state-dir", "--migrations-dir"):
    if i + 1 >= len(argv):
        print(f"ERROR: {arg} requires a path argument", file=sys.stderr)
        sys.exit(1)
    ...
```

### R8-003 · RISK · `package_verifier.py` has zero unit test coverage

**Pattern:** #31 Regression Blind Spot
**File:** `proxmox-bootstrap/package_verifier.py` (entire file)

**Description:**
`test_migration_manager.py` covers `migration_manager.py` and `bootstrap_state.py` thoroughly. `package_verifier.py` has no corresponding test file and no coverage.

The untested surface includes:
- `_collect_files()` — the include/exclude boundary logic, including the self-exclusion of `package-descriptor.json` itself
- `compute_hash()` — the deterministic hash algorithm (path-then-content feed)
- `stamp()` — write path, field names, descriptor format
- `verify()` — diff output, mismatch detection, exit-code mapping (0/1/2)

The self-exclusion of the descriptor (`if p.resolve() == descriptor_resolved: continue`) is the most critical correctness property of the entire verifier — if it ever regresses, the stamp operation enters an infinite logical loop (hash depends on content of file that's being written). This property has no automated test.

**Recommended fix:** Add `tests/unit/test_package_verifier.py` covering at minimum:
- `_collect_files()` excludes `package-descriptor.json` itself
- `_collect_files()` excludes `__pycache__/` and `.json` files
- `compute_hash()` is deterministic across two calls with the same file set
- `verify()` returns 2 when descriptor is absent
- `verify()` returns 1 when a file is modified after stamping
- `verify()` returns 0 when package is unmodified

### R8-004 · OBSERVATION · Lock path is not parameterized when `--state-dir` is overridden

**Pattern:** #37 Hardcoded Environment
**Files:** `scripts/forge-quiesce.sh:26`, `scripts/forge-resume.sh:20`, `scripts/forge-migrate.sh:48`

**Description:**
`forge-quiesce.sh` and `forge-resume.sh` hardcode the lock path as `/var/lib/broodforge/migration.lock` with no override mechanism. `forge-migrate.sh` accepts `--state-dir` to override the state directory but calls `forge-quiesce.sh` without forwarding that parameter.

For the standard single-host deployment this is correct. For non-standard deployments (e.g., testing with `--state-dir /tmp/test-state`), the lock is always created at the system path regardless of the custom state dir, meaning:
1. Running `forge-migrate.sh --state-dir /tmp/test-state` still requires `/var/lib/broodforge` to be writable.
2. Two concurrent test migrations with different `--state-dir` values share the same lock.

**Impact:** Low for production (single deployment, fixed paths). Moderate for testing and non-standard deployments.

**Recommended fix:** Add a `--lock-dir` parameter to `forge-quiesce.sh` and `forge-resume.sh`, defaulting to `/var/lib/broodforge`. Have `forge-migrate.sh` forward `--lock-dir ${STATE_DIR}` when `--state-dir` is overridden. Alternatively, document the current behavior explicitly in the help text.

### R8-005 · RISK · `package-descriptor.json` is absent — stamp ceremony not yet run

**Pattern:** #38 Package Integrity Self-Verification
**File:** `proxmox-bootstrap/package-descriptor.json` (does not exist)

**Description:**
`package_verifier.py` and `forge-stamp-version.sh` provide the complete infrastructure for package integrity verification. However, `proxmox-bootstrap/package-descriptor.json` does not exist in the repository.

Per pattern #38: "A deployable package with no `package-descriptor.json` is a RISK." The integrity baseline has not been established for the Phase 1.N additions. Running `forge-verify-package.sh` on a clone of this repo today returns exit code 2 ("descriptor not found").

This is the expected state for code that has never been stamped — the tooling to fix it is precisely what was just added. But it remains a gap: until `bash scripts/forge-stamp-version.sh` is run and the resulting `package-descriptor.json` is committed, the project is in an unverifiable state.

**Recommended fix:** Run `bash scripts/forge-stamp-version.sh`, verify the output, and commit `proxmox-bootstrap/package-descriptor.json` as part of completing Phase 1.N.

### R8-006 · OBSERVATION · Dead `clock` parameter in `discover_migrations()` signature

**Pattern:** #31 Regression Blind Spot (dead parameter variant)
**File:** `proxmox-bootstrap/migration_manager.py:223`

**Description:**
`discover_migrations()` accepts `clock: Callable[[], datetime]` but never calls `clock()` in its body. The parameter is forwarded from `run_migrations()` (which DOES use it for `ran_at` stamping), but the forwarding is a no-op — removing `clock` from `discover_migrations`'s signature would not change any behavior.

The test suite passes `clock=self._fixed_clock()` to `run_migrations()`, which passes it to `discover_migrations()`, where it is silently ignored. As a result, the clock injection for `ran_at` is implemented but not exercised by any assertion (no test checks the `ran_at` field against the fixed clock value).

**Impact:** None functional. But the parameter creates a misleading contract — callers must provide `clock` to `discover_migrations()` despite it having no effect there.

**Recommended fix (option A):** Remove the `clock` parameter from `discover_migrations()`. `run_migrations()` already has the parameter and uses it correctly.

**Recommended fix (option B):** If the parameter is forward-looking (anticipating a future need to timestamp discovery events), add a comment saying so. Add a test asserting `record["ran_at"] == "2026-06-09T00:00:00Z"` to exercise the clock injection.
